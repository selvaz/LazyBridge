The real problem

Frameworks solve the wrong complexity.

Most LLM frameworks wrap provider APIs but add orchestration overhead in the process — manual dispatch loops, message list wrangling, glue code between agents. The mental model grows with every feature you add. LazyBridge is built to reverse that.

Zero noise in practice

The same model, from a single call to a full pipeline.

One class. One API. Regardless of complexity or provider.

Python 
from lazybridge import LazyAgent

answer = LazyAgent("anthropic").text("What is the capital of France?")
print(answer)

2 lines. No setup.

No client, no message list, no content extraction. Swap "anthropic" for "openai" — nothing else changes.

Python 
from lazybridge import LazyAgent, LazyTool

def get_weather(city: str) -> str:
    """Get current weather for a city."""
    return f"{city}: 22°C, sunny"

result = LazyAgent("anthropic").loop(
    "What's the weather in Rome and Paris?",
    tools=[LazyTool.from_function(get_weather)],
)
print(result.content)

Schema generated automatically from type hints + docstring. No JSON dict.

LazyTool.from_function() introspects the signature and docstring at call time. No decorator, no schema class, no manual parameter mapping.

Raw SDK equivalent: ~45 lines, including a while True loop you have to write yourself.

The composition model

Everything becomes a tool.

LazyBridge has one unifying rule. Every unit of work — a function, an agent, a session, a full pipeline — can become a tool. This is the composition model.

Python 
# A function
tool = LazyTool.from_function(my_fn)

# An agent
tool = agent.as_tool("researcher", "researches a topic")

# A session of agents (parallel)
tool = session.as_tool("market_intel", "...", mode="parallel")

# A pipeline (chain)
tool = session.as_tool("pipeline", "...", mode="chain")

One rule. Infinite composition.

Compose a tool into another agent's loop. Nest a pipeline inside a chain. Wrap a parallel research session as a single callable. The unit is always the same — which means the mental model never grows.

No special orchestration classes. No pipeline DSL. No graph builder.

Designed to be read

The framework should not be louder than the system.

🤖

LLMs writing

Every pattern is explicit. No hidden state. LazyContext, LazyStore, Memory are all named and composable. Code written by an AI is as readable as code written by a human.

Everyone reading

The code IS the mental map. Whether you wrote it, an AI wrote it, or you inherited it — LazyBridge pipelines read like system design documents.

Tools are portable

Observability

Full tracing. Zero infrastructure.

Every model call, tool invocation, and loop step is recorded in an embedded SQLite event log — automatically, with no setup. Query it from Python. Export it. Filter by agent, event type, or timestamp.

No cloud account. No OTel collector. No API key. No per-event billing. The event log lives next to your code, works offline, and costs nothing.

vs LangSmith Cloud account + per-event pricing. LazyBridge: one SQLite file, always free.
vs OTel Requires a collector infrastructure. LazyBridge: zero external dependencies.
Python
from lazybridge import LazyAgent, LazySession, Event

# tracking is on by default — nothing to configure
sess = LazySession(db="pipeline.db", tracking="verbose")

researcher = LazyAgent("anthropic", name="researcher", session=sess)
writer     = LazyAgent("openai",    name="writer",     session=sess)

# ... run your pipeline ...

# Query the log from Python
all_events    = sess.events.get()
tool_calls    = sess.events.get(event_type=Event.TOOL_CALL)
writer_events = sess.events.get(agent_id=writer.id)

for ev in tool_calls:
    print(ev["timestamp"], ev["agent_name"], ev["data_json"])
