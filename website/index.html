The primitive

One interface. Four levels of complexity.

The same contract holds whether you are calling a function, an agent, or a nested multi-agent pipeline. What changes is only what is inside.

Python 
# 1. A plain function
def summarize(docs: str) -> str:
    """Summarize a collection of documents."""
    return LazyAgent("anthropic").text(f"Summarize: {docs}")

# 2. Wrap as a tool — no changes to the function
tool = LazyTool.from_function(summarize)

# 3. Compose into an agent loop — tool is a first-class argument
agent = LazyAgent("anthropic")
result = agent.loop("Summarize these three research papers", tools=[tool])

# 4. Expose the agent as a tool — nest it into a pipeline
agent_tool = agent.as_tool("summarizer", "Summarizes a set of documents")
pipeline = LazySession().as_tool(
    "research_pipeline", "Fetch → summarize → report",
    mode="chain",
    participants=[fetch_tool, agent_tool, report_writer],
)

What stays the same

summarize, tool, agent_tool, and pipeline all look identical to their consumers. Pass to tools=[...] or call .run() — the shape never changes.

What changes

Only the contents of each unit. Step 1 is a function. Step 4 is three nested components across two providers. The calling convention is unchanged throughout.

No rewrite cost

Upgrading a function to an agent to a pipeline does not require rewriting the callers. The interface is the same at every level.

from langgraph.graph import StateGraph, END
from typing import TypedDict

class State(TypedDict):
    messages: list
    market_data: str
    risk_profile: dict

def fetch_node(state: State) -> State: ...
def risk_node(state: State) -> State: ...
def report_node(state: State) -> State: ...

graph = StateGraph(State)
graph.add_node("fetch",  fetch_node)
graph.add_node("risk",   risk_node)
graph.add_node("report", report_node)
graph.add_edge("fetch",  "risk")
graph.add_edge("risk",   "report")
graph.add_edge("report", END)
graph.set_entry_point("fetch")
app = graph.compile()

The composition model

Every node has the same shape, regardless of what is inside.

A single agent, a parallel session of three agents, and a full nested pipeline all expose the same interface. From outside the box: indistinguishable.

Same shape at every level

market_intel contains three parallel agents. From outside it is indistinguishable from fetch_agent, which is a single LazyAgent. Both are called the same way.

Nesting is free

Any node can become a participant in any chain or parallel session. No adapter layer, no wrapping class, no DSL change when promoting a unit to a higher level.

One rule. No cap.

Every unit of work — function, agent, session, full pipeline — exposes the same interface. The mental model does not grow as complexity grows.

What it removes

The overhead, not the capability.

These are the layers LazyBridge eliminates. Removing them does not reduce what you can build — it reduces what you have to carry while building it.

  • ✔
    Adapter layers between providers

    One string switches provider. No format translation, no separate client instances.

  • ✔
    Orchestration-specific DSLs

    No graph builders, no node type declarations, no edge definitions, no compile steps.

  • ✔
    Heterogeneous abstraction levels

    Functions, agents, and pipelines are the same unit. One abstraction level throughout.

  • ✔
    Manual dispatch loops and message wrangling

    No while True loops, no manual tool dispatch, no message list management.

  • ✔
    Rewrite cost when upgrading a component

    A function promoted to an agent to a pipeline does not require rewriting its callers.

  • ✔
    Translation logic between providers

    Mix Anthropic, OpenAI, and Google in one pipeline. No serialization glue between steps.

Built for both authors

Code that reads the same whether a human or an LLM wrote it.

🧑‍💻

Humans writing

Type hints become tool schemas. Functions become agents. Pipelines compose like data structures. No JSON dicts, no dispatch loops, no message list wrangling.

🤖

LLMs writing

Every pattern is explicit. No hidden state. LazyContext, LazyStore, Memory are all named and composable. Code written by an AI is as readable as code written by a human.

📖

Everyone reading

The code IS the mental map. Whether you wrote it, an AI wrote it, or you inherited it — LazyBridge pipelines read like system design documents.

💾

Tools are portable

tool.save("my_tool.py") generates a plain Python file — readable, editable, version-controllable. LazyTool.load() reloads it in any process or machine, fully operational.

Is this right for you?

LazyBridge is not for every problem.

These are honest answers to when to use it and when not to.

✔

LazyBridge is right for you if…

  • ▸ You are building multi-agent systems with nested composition.
  • ▸ You are mixing providers (Anthropic, OpenAI, Google) in one pipeline.
  • ▸ You want components that compose without adapter layers.
  • ▸ You want LLM-generated code to be as readable as human-written code.
  • ▸ You want the mental model to stay constant as complexity grows.
→

Use something else if…

  • ▸
    You need durable execution and fault tolerance → use Temporal
  • ▸
    You need retries, checkpointing, or exactly-once guarantees → use Temporal
  • ▸
    Your team needs a visual workflow editor → use LangGraph Studio or similar
  • ▸
    You need enterprise runtime guarantees or SLA-backed infrastructure → use a managed orchestration platform

          native_tools=[NativeTool.WEB_SEARCH])

          native_tools=[NativeTool.WEB_SEARCH])

          native_tools=[NativeTool.WEB_SEARCH])
market_intel_tool = market_sess.as_tool("market_intelligence", "...", mode="parallel")

          output_schema=RiskProfile)

Every model call, tool invocation, and loop step is recorded in an embedded SQLite event log — automatically, with no setup. Query it from Python. Export it. Filter by agent, event type, or timestamp.

No cloud account. No OTel collector. No API key. No per-event billing. The event log lives next to your code, works offline, and costs nothing.

vs LangSmith Cloud account + per-event pricing. LazyBridge: one SQLite file, always free.
vs OTel Requires a collector infrastructure. LazyBridge: zero external dependencies.
Python
from lazybridge import LazyAgent, LazySession, Event

# tracking is on by default — nothing to configure
sess = LazySession(db="pipeline.db", tracking="verbose")

researcher = LazyAgent("anthropic", name="researcher", session=sess)
writer     = LazyAgent("openai",    name="writer",     session=sess)

# ... run your pipeline ...

# Query the log from Python
all_events    = sess.events.get()
tool_calls    = sess.events.get(event_type=Event.TOOL_CALL)
writer_events = sess.events.get(agent_id=writer.id)

for ev in tool_calls:
    print(ev["timestamp"], ev["agent_name"], ev["data_json"])
